# Лабораторна робота: REST API (Notes.FX Backend)

## Тема
Проєктування та реалізація REST API для сервісу нотаток з використанням Django та Django Ninja.

## Мета роботи
Мета цієї лабораторної роботи полягає в ознайомленні з побудовою REST API, організацією шарів застосунку, використанням DTO/Schema для вхідних і вихідних даних та документуванням API через Swagger UI.

## Опис архітектури
Застосунок побудований на Django з використанням Django Ninja. Код розділено по модулях `apps/users`, `apps/notes` та `apps/community`, а роутинг API здійснюється в `notesFx/urls.py` через підключення `NinjaAPI` та відповідних роутерів. Доступ до бази даних реалізовано через Django ORM, моделі зберігаються у `apps/*/models.py`.

## Реалізація API (ключові модулі)
Основна логіка роботи з нотатками, категоріями, тегами, файлами та історією змін реалізована в `apps/notes/api.py`. Реєстрація користувачів, логін і refresh-token описані в `apps/users/api.py`. Функціонал публічної стрічки, коментарів і лайків розміщено в `apps/community/api.py`.

## Приклади коду (фрагменти у звіті)
У якості прикладу DTO/Schema використовується `NoteCreateWithPasswordSchema` у `apps/notes/schemas.py`, де описані поля заголовку, контенту, категорії, тегів, публічності та пароля. Реалізація методу контролера показана на прикладі `create_note` у `apps/notes/api.py`, де створюється нотатка, встановлюється пароль за потреби, додаються теги та повертається створений об’єкт або помилка.

## Приклади REST ендпоїнтів
У застосунку реалізовано типові REST ендпоїнти, такі як `POST /api/auth/register` для реєстрації, `POST /api/auth/login` для логіну, `GET /api/notes/notes` для отримання списку нотаток, `POST /api/notes/notes` для створення нотатки та `GET /api/community/feed` для публічної стрічки.

## Архітектура та доступ до даних
Доступ до даних побудований на Django ORM з моделями `Note`, `Category`, `Tag`, `Reminder` та іншими. API реалізовано через Django Ninja, роутери рознесено по модулях. DTO/Mapping виконуються через Pydantic/Schema класи, а частина відповіді мапиться вручну, наприклад у функції `note_to_response`. Валідація запитів виконана в контролерах, зокрема у `create_reminder`, `upsert_mood` та інших методах.

## Висновки
Під час виконання лабораторної роботи були отримані практичні навички реалізації REST API, опису DTO/Schema, підключення авторизації та документування API. Робота продемонструвала повний цикл проєктування і документування бекенду для сервісу нотаток.

## Контрольні запитання

### 1) Порівняйте REST, GraphQL та gRPC. У яких випадках доцільно використовувати кожен із них?
REST доцільний для типових CRUD API з широкою сумісністю та можливістю кешування на рівні HTTP. GraphQL варто використовувати, коли клієнт потребує гнучких запитів і мінімізації зайвих даних завдяки точному вибору полів. gRPC є доцільним для високопродуктивної внутрішньої взаємодії сервісів, зокрема коли потрібен стрімінг і низькі накладні витрати.

### 2) Що таке проблема Over-fetching та Under-fetching даних? Як GraphQL вирішує ці проблеми порівняно з REST?
Over-fetching означає повернення більшої кількості даних, ніж потрібно клієнту, а Under-fetching — ситуацію, коли для повного набору даних потрібно робити додаткові запити. GraphQL вирішує ці проблеми тим, що клієнт сам визначає точний набір полів у запиті, отримуючи все необхідне за один раз.

### 3) Що таке Protocol Buffers (Protobuf)? Які його переваги перед JSON?
Protobuf — це бінарний формат серіалізації даних зі строгою схемою. Його переваги полягають у меншому розмірі повідомлень, швидшому парсингу та чітко визначеному типізованому контракті між клієнтом і сервером.

### 4) Як забезпечується версіонування API у REST та GraphQL?
У REST версії зазвичай закладаються у URL (наприклад `/v1/`), у заголовки або через контент-неймспейси. У GraphQL версіонування реалізують через еволюцію схем із депрекейтом полів та додаванням нових без ламання існуючих клієнтів.

### 5) Поясніть концепцію Idempotency (ідемпотентності). Які методи/операції мають бути ідемпотентними?
Ідемпотентність означає, що повторний виклик операції не змінює результат після першого виконання. Ідемпотентними мають бути методи `GET`, `PUT` та `DELETE`, тоді як `POST` зазвичай не є ідемпотентним.

### 6) Яка роль Schema Definition Language (SDL) у GraphQL або .proto файлів у gRPC?
SDL у GraphQL описує типи, запити й мутації та виступає контрактом між клієнтом і сервером. Файли `.proto` у gRPC описують повідомлення і RPC-методи та є контрактом, з якого генерується клієнтська і серверна частина.

## Система оцінювання (ТН)
Усього за лабораторну роботу можна отримати 20 балів. Налаштування доступу до даних (ORM/Repository) оцінюється у 4 бали, архітектура (services, DI, чистота коду) — у 6 балів, коректна реалізація API — у 6 балів, використання DTO/Mapping — у 2 бали, наявність інтерфейсу для тестування (Swagger/GraphiQL) — у 2 бали, а бонус у 2 бали надається за реалізацію валідації вхідних даних.

### Приклади з проєкту для критеріїв
Приклади підтвердження критеріїв з проєкту такі: ORM підтверджується моделями у `apps/notes/models.py` (наприклад `Note`, `Category`, `Tag`), архітектура — модульною структурою та роутерами в `apps/*/api.py` з підключенням `JWTAuth` у `notesFx/urls.py`, коректна реалізація API — ендпоїнтами на кшталт `@router.post("/notes")` і `@router.get("/notes/{note_id}")`, DTO/Mapping — схемами в `apps/notes/schemas.py` та мапінгом у `note_to_response`, інтерфейс тестування — Swagger UI через `NinjaAPI(..., docs_url="/docs")`, а валідація — перевірками у `upsert_mood`, `create_reminder`, `set_note_password`.
